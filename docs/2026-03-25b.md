## Top 3 Recommendations

### #1 — "Start Your Season" Wizard: Fix the Cold Start Before It Kills Beta ⚡ UNANIMOUS
**What:** Right now, a new user's flow is: Google OAuth → `FamilySetup` (family name + create/join) → empty dashboard. That's it. The `FamilySetup.tsx` component creates the family then drops them on `/dashboard` with zero events, zero sports seasons, zero family members beyond themselves. A dashboard widget called `TodayTimeline` shows "No events today." A parent sees a blank screen and closes the app. Done. Churned forever.

**The fix — a 5-step "Start Your Season" modal wizard that fires on first login:**
1. **Add your kids** — name + color (the `User` model is already set up for family members, invite code flow exists)
2. **Pick a sport** — dropdown + custom entry, creates `SportSeason` record
3. **Scan your schedule** — trigger `ScheduleScanFlow.tsx` right here. It already exists. AI parses the photo and creates all `SportEvent` records in one shot.
4. **Who's driving?** — two toggles: "I'm available to drive" → creates `DriverAvailability` records for weekday/weekend
5. **You're set!** — redirect to dashboard, which now has events, a season, and driver assignments.

**Why this is #1:** The `ScheduleScanFlow.tsx` component is already production-ready. The schema is fully in place. This is a **stitching** problem — nobody connected these existing pieces into a first-time setup flow. Every beta user who bounced after signup likely bounced because of this exact gap. This can be built in a sprint.

**The data signal:** `AiUsage.scheduleScansCount` is tracked. If beta families have low scan rates, the scanner isn't being discovered. The scanner needs to be *the first thing they do*, not buried in Sports → Seasons → scroll down → find camera icon.

**Gap identified:** `app/(auth)/signup/page.tsx` is literally a redirect to `/login`. There's no sports-specific onboarding path. The entire onboarding is: "Sign in with Google." KinSync has no opportunity to collect sports context before dumping the user on an empty dashboard.

**Effort:** Medium | **Impact:** Critical (activation → retention → word-of-mouth)

### #2 — Per-Event RSVP: TeamSnap's #1 Sticky Feature, Missing from KinSync
**What:** TeamSnap's #1 retention mechanism is the per-game RSVP. Before every game, parents get a message: *"Is [child] attending Thursday's game?"* One tap. Coach sees attendance. Parent feels organized. KinSync has **zero equivalent**.

The current schema has `DriverAvailability` (weekly recurring windows) and `PickupAssignment` (who drives), but **no model for "Is [child] actually going to this game?"** That's a different question. A kid can be injured, grounded, at a conflict. The driver assignment is meaningless without knowing the player is showing up.

**What needs to be built:**

**Schema addition (small):**
```prisma
model SportEventRSVP {
  id           String     @id @default(cuid())
  sportEventId String
  sportEvent   SportEvent @relation(...)
  userId       String     // the parent responding
  childId      String     // the child being RSVP'd for
  status       String     @default("PENDING") // PENDING | GOING | NOT_GOING | MAYBE
  note         String?
  updatedAt    DateTime   @updatedAt
  createdAt    DateTime   @default(now())

  @@unique([sportEventId, childId])
  @@index([sportEventId])
  @@index([userId])
}
```

**Push notification flow (48hrs before game):** "Is Luke going to Thursday's game vs. Jefferson? 👍 Going / 👎 Not Going" → one-tap response.

**In-app surface:** On the Sports → Schedule tab, each upcoming game shows RSVP status badges per child. Red = Not Going. Green = Going. Gray = Pending. Parents can see at a glance who's confirmed.

**Why this matters:** This feature makes KinSync interactive, not passive. Right now KinSync is a calendar viewer. RSVP makes it a *coordination tool*. Parents open it to respond. Coaches can be given a read-only share link to see attendance (via existing `ShareLink.resourceType: SPORT_EVENT`). This is the fastest path to "my whole team uses this."

**Note:** This is NOT building a team app. The RSVP is between the parent and their own family's KinSync — "My child is/isn't going." It surfaces to a coach only if the parent shares. That's the correct posture.

**Effort:** Medium | **Impact:** High (weekly active use, coach share = team viral loop)

### #3 — Define PRO_PLUS Before You Hit the Revenue Ceiling
**What:** The `SubscriptionTier` enum has `FREE`, `PRO`, `PRO_PLUS`. The billing routes are live (`/api/billing/checkout`, `/api/billing/portal`, `/api/billing/verify`). Stripe is integrated. But `PRO_PLUS` appears almost nowhere in the business logic — practically everything gates on `PRO`. This is a pricing structure with a ghost tier.

**The definition:** Make PRO_PLUS the **Multi-Sport Power Family** plan. Price it at $99/yr. Target families with 3+ active `SportSeason` records simultaneously (KinSync can detect this — `SportSeason.status = "ACTIVE"` count per family). Features exclusive to PRO_PLUS:

| PRO_PLUS Feature | Why It Works | Schema Hook |
|---|---|---|
| **Multi-kid Game Day Mode** | Today shows ALL active kids' games, not just one | `SportSeason.childId` → aggregate |
| **Cross-season conflict detection** | "Luke has practice AND Logan has a game — same time, same driver" | Existing `Event` + `PickupAssignment` query |
| **Tournament weekend planning** | Multi-day event mode for travel tournaments | `SportEvent.eventType = TOURNAMENT` already exists |
| **Per-event RSVP (from #2)** | PRO_PLUS gets the RSVP feature first | New `SportEventRSVP` model |
| **Season history archive** | Keeps completed seasons browsable forever (FREE archives after 90 days) | `SportSeason.status = ARCHIVED` |

**Why define it now:** Matt is targeting beta testers at $39/yr for life. If PRO_PLUS doesn't exist at launch, the lifetime beta users will never pay more. Define it now, beta users get `PRO` (mid-tier). Power families who want more see the `PRO_PLUS` upgrade path immediately.

**Revenue model insight:** The `AiUsage` table tracks `scheduleScansCount`, `navigatorCheckinsCount`, etc. At 10k families, even if only 15% (1,500 families) have 3+ active sports seasons, and 30% of those upgrade to PRO_PLUS at $99/yr, that's **$44,550/yr incremental ARR from this tier alone** — just from defining it.

**Effort:** Low (schema + billing config) | **Impact:** High (revenue ceiling removed)

## Big Bets (Worth Exploring This Quarter)

### The Coach Share Link (Team-Side Distribution)
The `ShareLink` model supports `resourceType: SPORT_EVENT` and `SPORT_SEASON`. Right now, sharing goes family-to-family or family-to-grandparent. The untapped direction: **family-to-coach**. A parent generates a "Follow Luke's season" link. Coach opens it, sees: schedule, RSVP status, upcoming games. Coach bookmarks it. Then coach tells OTHER parents on the team: "Check Luke's KinSync link — it shows who's coming Thursday." That's the team-side distribution loop. KinSync doesn't have to build a coach portal — just make the share links impressive enough that coaches share them.

### Infinite Campus × Sports: The Cross-Context View Nobody Does
`InfiniteCampusConnection` is already built and live in settings. IC syncs school assignments and schedules as `Task` records with `source: "INFINITE_CAMPUS"`. But IC data and Sports data never talk to each other. The insight: **"Luke has a game Thursday AND a history test Friday"** — that's a parent's nightmare week, and KinSync is the only app that can see both. Build a "Conflict Radar" view in the Planner: IC tasks + SportEvents on the same timeline. Highlighted in amber when a game-night lands before a test-day. Zero new schema work — it's a query joining `Task WHERE source = 'INFINITE_CAMPUS'` with `SportEvent`.

### The Public Gear Checklist (SEO-Driven Content Play)
`PublicList` model supports `slug`, `isPublished`, `viewCount`. This is a public-facing list that can be SEO-indexed. The play: Matt publishes 10 "equipment checklists" pre-built — Soccer Tournament Bag, Baseball Weekend Camp Gear, Basketball Tryout Essentials. Parents Google "soccer tournament packing list for kids" → land on KinSync's public list → see the KinSync watermark → sign up. `PublicList.completionMode = SHARED` means the whole family can check items off together on game morning. This is a **content marketing + viral loop** with no engineering cost — the model is already built and deployed (`app/(public)/lists/[slug]/page.tsx` exists).

### `WeekPlan` × Sports: The AI-Powered Sports Week Briefing
`WeekPlan` + `WeekPlanItem` already exist (the Autopilot Week feature). `WeekPlanItemCategory` includes `SCHEDULE`, `LOGISTICS`, `CONFLICT`, `HEADS_UP`. But the Autopilot Week planner appears to pull from general calendar events — there's no evidence it's sports-aware. A `SPORT_GAME` category added to `WeekPlanItemCategory` would let the AI say: *"Luke has 3 games this week. Tuesday is away — leave 30 min earlier. Thursday conflicts with Sarah's work event. You have no driver for Friday's practice."* That's genuinely useful and sells the AI tier.

## Debate Round Highlights

**Atlas flagged (risky from Nova):** Defining PRO_PLUS as a hard tier before beta testing pricing is premature. Lock in the FEATURES, not the price. $99/yr for a beta launch could alienate the exact power-user families you need for word-of-mouth. **Elevation:** Gate PRO_PLUS features with a "coming soon for power families" banner during beta — qualify demand before charging for it.

**Nova flagged (risky from Prism):** The onboarding ribbon is correct but placement is wrong. A horizontal banner across the top of the dashboard competes with `TodayTimeline` and `WeatherWidget` for attention. Better: make the ribbon a **full-screen overlay on first login** (like Notion's workspace setup, or Headspace's onboarding). Don't let users skip to an empty dashboard. Force the first 3 setup steps.

**Prism elevated (underrated from Forge):** The cron health logging is boring infrastructure — but it's actually a UX feature. If Matt can see a settings page showing "Safety Check: last ran 4 min ago ✓ | Weekly Briefing: last ran 6 days ago ✓", he stops worrying about whether the app is working. That builds founder trust in the system, which affects what he ships next. Operational visibility is UX for the admin.

**Forge flagged (risky from Echo):** The per-event RSVP (from #2) requires a new table and push notification flow. The risk isn't schema size — it's notification fatigue. If KinSync sends an RSVP push 48hrs before every event (game + practice + tournament), that's potentially 4-6 pushes per week per family. Parents will mute them. **Fix:** RSVP push only for `SportEventType = GAME | TOURNAMENT | MEET`. Skip practices. Set a default preference `rsvpNudgeEnabled: true` in `NotificationPreference`.

**Echo elevated (underrated from Atlas):** The Infinite Campus × Sports "Conflict Radar" is the most defensible competitive moat in this memo. No competitor — not TeamSnap, SportsEngine, Band, or Google Calendar — sees BOTH school assignments AND sports schedules in one view. IC integration is already live. This is a SQL JOIN, not a feature. It should ship before the coach portal conversation ever starts.
